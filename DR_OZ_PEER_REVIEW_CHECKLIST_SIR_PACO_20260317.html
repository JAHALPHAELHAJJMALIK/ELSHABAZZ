▼ Critical Gaps — Remediation Required Before Deployment
⚠  Gaps — Must Address
GAP 1 — Non-CS Proc: Hardcoded Test Dates (CRITICAL) DECLARE block in NON_CS proc has STARTDATE := '2025-01-01' and ENDDATE := '2026-03-10' uncommented. Proc will ignore passed parameters entirely. Comment these lines out before any production call.
GAP 2 — Modifier Codes: NULL in Non-CS Proc NULL AS "Modifier Codes" is hardcoded. The spec requires modifiers for all datasets. For ABA/BHT credential-modifier crosswalk (Report 3) to work, mh.MOD_CODE_1–5 must be surfaced. Recommend uncommenting and using the CONCAT_WS approach (already present as a comment in both procs).
GAP 3 — ABA Code Range Incomplete Spec references the full ABA range 97151–97158. Codes 97154, 97157, 97158 are absent from the Non-CS proc WHERE clause and CASE statement. Confirm with clinical/coding if these are billable by CHG providers.
GAP 4 — Short-Term Post Hospitalization Housing Misclassified The spec lists Short Term Post Hospital under "14. Recuperative Care" but the CS proc CASE sends it to the ELSE bucket → '8. Integrated Community Supports'. Add an explicit WHEN:
WHEN COMMUNITY_SUPPORTS_SERVICE = 'Short-Term Post Hospitalization Housing' THEN '14. Recuperative Care'
GAP 5 — Housing Navigation & Sustaining Misclassified Spec places Housing Navigation and Housing Sustaining under "6. Housing Stabilization Services." Currently both fall to ELSE → '8. Integrated Community Supports'. Add explicit WHENs for 'Housing Transition Navigation Services' and 'Housing Tenancy and Sustaining Services' → '6. Housing Stabilization Services'.
GAP 6 — Analytical / FWA Report Logic Not Built Both procs are raw population extracts. The Dr. Oz spec requires 20+ specific analytical reports (hospital overlap, provider time outliers, duration compliance, NEMT cross-reference, etc.). These are entirely absent from the current code. Sir Paco's procs provide the data foundation — the report logic is a separate build increment.
GAP 7 — HCPCS/CPT in CS Proc from Auth, Not Claim CS proc uses auth.HCPC_CODE_SEARCH (from DIM_AUTHORIZATIONSERVICE) rather than the claim-level service code. The commented-out mh.HCP_CODE is the claim-level code. Confirm with Dr. Oz whether the auth code or the billed claim code is the intended data element. These may differ for modified/substituted services.
GAP 8 — Admit Date Missing "Admission/discharge dates" only surfaces mh.DISCHARGE_DATE. mh.ADMIT_DATE is available in FACT_MEMBERHISTORY. For hospital overlap reports (CBAS, Personal Care, Recuperative Care), the admit date is essential. Add TO_DATE(mh.ADMIT_DATE::VARCHAR,'YYYYMMDD') AS "Admission Date" as a separate column, or confirm the spec intends only discharge.
▼ Observations & Recommendations (Non-Blocking)
ⓘ  Notes — Review with Sir Paco
Unused PAYTO JOIN Both procs JOIN DIM_PROVIDERSPECIALTY AS payto on mh.PAYTOPROVIDERSK but select nothing from it. This is an unnecessary INNER JOIN that restricts rows and adds cost. If payto data is not needed in the output, remove the JOIN. If payto NPI/Tax ID is needed (for NEMT or nursing billing), add the appropriate column.
Auth JOIN Grain Risk (CS Proc) INNER JOIN on DIM_AUTHORIZATIONSERVICE means any paid CS claim without a matched AUTHORIZATIONSERVICESK will be silently excluded. Recommend auditing for NULL AUTHORIZATIONSERVICESK in FACT_MEMBERHISTORY where COMMUNITY_SUPPORTS_SERVICE IS NOT NULL and converting to LEFT JOIN if gaps are found.
DISTINCT May Collapse Service Lines Both procs use SELECT DISTINCT across a wide column set. Verify that the intended grain is one row per unique claim line. If two service lines on the same claim share all selected fields except DOS, DISTINCT could still keep both rows — confirm desired output grain with Dr. Oz team.
Transitional Rent Missing from CS Service Type CASE 'Transitional Rent' is a valid COMMUNITY_SUPPORTS_SERVICE value per STEP88 and the Snowflake definition. It is not listed in the CS proc CASE — it will fall to '8. Integrated Community Supports'. Confirm if this is intentional or if a mapping to "6. Housing Stabilization Services" is needed (per spec intent).
Modifier Codes — CS Proc Surfaces Auth Modifier Only CS proc outputs auth.HCPC_MODCODE_SEARCH_1 (one auth-level modifier). The commented-out alternative uses CONCAT_WS(',', mh.MOD_CODE_1 ... MOD_CODE_5) which is the full claim-level modifier string and would better support the credential-modifier crosswalk reports. Recommend uncommenting for consistency with Non-CS approach once NULL issue is resolved.
Output RETURNS TABLE — VARCHAR(33) for Service Type The RETURNS TABLE definition specifies "Community Support Service Type" VARCHAR(33) in the CS proc and "Service Type" VARCHAR(33) in Non-CS proc. The value '5. Early Intensive Developmental & Behavioral Intervention' is 54 characters — will exceed VARCHAR(33) and cause a runtime error. Increase to VARCHAR(100) minimum.
▼ Overall Verdict
✓  Peer Review Summary — El Hajj Malik Jahwaca | 03/17/2026
Sir Paco's two stored procedures provide a solid data foundation — the member demographic block, provider fields, paid claim filtering, CS/non-CS partitioning, and date handling are all well-constructed and consistent with the CHGSD Snowflake standards.

However, the code represents raw population extracts only. The Dr. Oz specification requires 20+ analytical and FWA-detection reports built on top of this foundation — none of which are present in the current submission. This should be understood as Phase 1 of a multi-phase build.

8 gaps require action before deployment, with GAP 1 (hardcoded test dates in the Non-CS proc) being the only critical production blocker. GAP 4 and GAP 5 (service type misclassification) and GAP 6 (missing ABA codes) directly affect the accuracy of Dr. Oz's analytical output and should be resolved in the next code iteration. The VARCHAR(33) limit on the Service Type return column (observation 6) will cause a runtime error and should be addressed before the proc is called.
