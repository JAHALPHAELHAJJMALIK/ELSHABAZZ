Report Purpose
DHCS D-SNP Palliative Care — QA
Identifies D-SNP members newly enrolled in palliative care services within a quarterly reporting period. Two parallel tracks run sequentially: (1) Claims Track — qualifies members via ICD-10 Dx Z51.5 on HMOPROD_PlanData claims; (2) Auth Track — qualifies members via authorization referrals carrying the same diagnosis via QNXT referral / authdiag tables. Both tracks exclude members who had a palliative claim in a prior quarter. Final deliverable is a Race/Ethnicity stratified template (TEMPLATE_PALLIATIVE / CHPIV_TEMPLATE_PALLIATIVE) loaded to MS Excel via OLE DB for DHCS submission.

Source Tables, Views & Work Tables — Full Inventory
HMOPROD_PlanData · QNXT (SQLPROD02) — Transactional Source
dbo.claim
Source
Claim header. Key fields: claimid, memid, planid (→ LOB/BRAND), startdate, enddate. Pre-disqualify uses startdate < @RangeStartDate. Qualify uses enddate ≥ @RangeStartDate AND startdate ≤ @RangeEndDate.
dbo.claimdiag
Source
Claim-level diagnosis codes. Fields: claimid, codeid, sequence, poa (Present on Admission), diagtype. Inner ROW_NUMBER() subquery partitions by claimid, orders by sequence ASC + codeid DESC. All sequences qualify for Z51.5 — no ROWis = 1 filter applied.
dbo.DiagCode
Source
ICD-10 code description lookup. LEFT JOIN from claimdiag ON codeid = codeid. Provides 'diag descr' for QA output columns and confirms Z51.5 is active via opening diagnostic SELECT.
dbo.referral
Source
Auth referral header for Auth track. Fields: referralid, authorizationid, memid, seendate, effdate, termdate, planid. Date window uses seendate BETWEEN @ClockStart AND @ClockStop. Crossfooted guard: ISNULL(termdate, GETDATE()) >= effdate.
dbo.authdiag
Source
Auth-level diagnosis codes. INNER JOIN to referral ON referralid. Fields: referralid, diagcode. Z51.5 filter: UPPER(LTRIM(RTRIM(ISNULL(ad.diagcode,'')))) LIKE '%Z51.5%'.
INFORMATICS · dbo (SQLPRODAPP01) — Analytics & Output
uvw_LINE_OF_BUSINESS
View
planid → LINE_OF_BUSINESS + BRAND. Joins HMOPROD_PlanData.dbo.benefitplan + dbo.program. Used as INNER JOIN on claim.planid and referral.planid for all LOB/BRAND filtering. Cardinality note: LINE_OF_BUSINESS WHERE filter is FINAL per script comment.
uvw_BASELINE_MEMBERSHIP
View
memid → CIN crosswalk with member demographic base. Used in Phase 3 to resolve CIN from memid for the EnrollmentManager race/ethnicity join chain.
Ethnicity_EICandOMBDesc_XWalk

QA_PALLIATIVE_CLAIM
Perm Output
CHGSD claims track permanent output. Contains claimid, memid, diag descr, InclusionExclusionType = 'PALLIATIVE'. Written via setclaim CTE SELECT INTO. Dropped and rebuilt each quarterly run.
QA_PALLIATIVE_AUTH
Perm Output
CHGSD auth track permanent output. Contains referral/auth-sourced palliative members with authorizationid, memid, InclusionExclusionType. Written via SETAUTHREF CTE SELECT INTO. Dropped and rebuilt each quarterly run.
TEMPLATE_PALLIATIVE

dbo.healthcoverage

Enrollment coverage spans. Fields: ClientIndexNumber, FileId, BenefitBeginDate, BenefitEndDate. Date window: BenefitEndDate >= @ClockStart AND BenefitBeginDate <= @ClockStop.

Temp Tables — Session Work Tables (All #Prefix)
#PREDISQUALIFYING_CLAIMS
Temp
Prior-quarter palliative claim holders. All Z51.5 claims with startdate before @RangeStartDate. claimid CLUSTERED + memid NONCLUSTERED indexes. DUP validation SELECT runs immediately after build.
#QUALIFYING_CLAIMS
Temp
Current-period new palliative claim holders. Z51.5 filter + date range + NOT IN #PREDISQUALIFYING_CLAIMS. Same clustered/nonclustered index pair. DUP validation SELECT after build.
#baselinememiso
Temp
UNION ALL of Auth + Claim members resolved to CIN via uvw_BASELINE_MEMBERSHIP. All 17 Race/Eth flag columns initialized to 0. DUP validation on memid after build. All sequential UPDATEs applied here.
#MemberEnrollmentEthnicityRace
Temp
CIN + OMBdesc pairs from EnrollmentManager join chain. Date-windowed to @ClockStart/@ClockStop. Used as source for all 17 flag UPDATEs on #baselinememiso.
#multiplerace
Temp
CIN + COUNT(DISTINCT OMBdesc) AS Numberrace (excluding Hispanic/Latino). Numberrace >= 2 triggers Race_Two_Or_More_Races = 1 UPDATE. Numberrace = 1 used in individual race flag UPDATEs.
Analyst Runbook — Quarterly Execution Steps
01 · @BRAND / @LOB Pre-Flight Check

Confirm which script version to run: QA_DHCS_DSNP_PALLIATIVE_… (CHGSD, @BRAND = 'CHGSD', @LOB = 'DSNP') or CHPIV_QA_DHCS_DSNP_PALLIATIVE_… (CHPIV, @BRAND = 'CHPIV', @LOB = 'CHPIV'). Verify @leadlag = -4 and @leadlagmonths = 3 produce the correct reporting quarter window. The opening diagnostic SELECT on uvw_LINE_OF_BUSINESS must return valid DSNP/CHPIV benefit plan rows before proceeding.

02 · Dx Z51.5 Code Confirmation

The second diagnostic SELECT on HMOPROD_PlanData.dbo.DiagCode WHERE codeid LIKE '%Z51.5%' confirms the code is active in the transactional system and returns the expected description 'Encounter for palliative care'. If this returns 0 rows, stop and investigate DiagCode table status before continuing.

03 · #PREDISQUALIFYING_CLAIMS DUP Validation

After the Pre-Disqualify phase writes to #PREDISQUALIFYING_CLAIMS, confirm the DUP validation SELECT returns 0 rows. Review the InclusionExclusionType COUNT — the PALLIATIVE count here represents prior-quarter palliative members who will be excluded from the current period. Note this count for comparison against the final qualified member count.

04 · #QUALIFYING_CLAIMS + setclaim CTE → QA_PALLIATIVE_CLAIM

After #QUALIFYING_CLAIMS is built and indexed, confirm its DUP validation returns 0 rows. The setclaim CTE writes INFORMATICS.dbo.QA_PALLIATIVE_CLAIM. Confirm the COUNT(DISTINCT memid) result makes sense relative to the pre-disqualify count and prior quarter's submission member count. A significant drop (>20%) warrants investigation of the @RangeStartDate window.

05 · SETAUTHREF CTE → QA_PALLIATIVE_AUTH

Auth track runs after Claims track completes. SETAUTHREF CTE uses referral + authdiag (Z51.5) + the same NOT IN #PREDISQUALIFYING_CLAIMS exclusion. @Authlob and date window (@ClockStart / @ClockStop) are verified at the AUTH parameter block. Confirm QA_PALLIATIVE_AUTH COUNT(DISTINCT memid). Auth count is typically lower than claims count — both are included in the UNION ALL for the template.

06 · #baselinememiso Build + Race/Eth UPDATEs

Phase 3 unions QA_PALLIATIVE_AUTH and QA_PALLIATIVE_CLAIM, resolves CIN, and applies 8 sequential UPDATE passes. After all UPDATEs, review the #baselinememiso SELECT — Race_Unknown and Race_Asked_But_No_Answer_with_Eth_Not_Hispanic counts should be reasonable. If Race_Unknown exceeds ~30% of total members, investigate the EthnicityIndustryCode data in EnrollmentManager or the Ethnicity_EICandOMBDesc_XWalk crosswalk for missing entries. DUP validation on memid confirms 0 duplicate rows.

07 · TEMPLATE_PALLIATIVE Final Output + MS Excel Refresh

The final SELECT writes INFORMATICS.dbo.TEMPLATE_PALLIATIVE (or CHPIV variant). Confirm the single-row output via the SELECT from TEMPLATE_PALLIATIVE. Open the DHCS submission MS Excel workbook and refresh the OLE DB connection to pull the new row. Validate: H Contract # = 'H4377/D-SNP', Reporting Period matches expected quarter dates, Member Count is reasonable vs prior quarter. Run both CHGSD and CHPIV scripts before populating the final DHCS submission form.

Senior Analyst Note (WCARR): The most common failure mode in this measure is an incorrect @leadlag / @leadlagmonths combination that shifts the date window. Always execute the parameter diagnostic SELECT first and confirm the reported BETWEEN dates match the target reporting quarter in the DHCS submission calendar. The pre-disqualification logic is equally critical — if #PREDISQUALIFYING_CLAIMS is empty when it should not be (e.g., after the first year of operation), the NOT IN exclusion has no effect and the qualified count will be higher than expected. Cross-validate against PROD_CHG_ODS.INFORMATICS.PreviousPalliativeMembers (Snowflake QA block, currently commented out) when in doubt.
Dual-Script Reminder: Both the CHGSD and CHPIV scripts must be executed before populating the final DHCS submission template. The CHPIV script targets INFORMATICS.dbo.CHPIV_TEMPLATE_PALLIATIVE — this is a separate permanent table from the CHGSD TEMPLATE_PALLIATIVE. Running only one script will leave one brand's submission data stale from the prior quarter.
Script Parameters — QA_DHCS_DSNP_PALLIATIVE_CLAIMS_AND_AUTHREFS
All parameters declared at the top of each script. Changes must be documented in the COMMENT(S) / CHANGE.LOG section with a C### reference per DUB C 2.0. @BRAND is the only parameter that differs between CHGSD and CHPIV scripts.
@BRAND
'CHGSD' | 'CHPIV'
Controls BRAND filter on uvw_LINE_OF_BUSINESS. 'CHGSD' = Community Health Group San Diego. 'CHPIV' = Community Health Plan of Imperial Valley. This is the ONLY parameter that differs between the two scripts.
@LOB
'DSNP' | 'CHPIV'
LINE_OF_BUSINESS LIKE filter. CHGSD script: 'DSNP'. CHPIV script: 'CHPIV'. Applied as FINAL FILTER per cardinality hierarchy guidance. Never NULL.
@PalliativeDx
nvarchar = 'Z51.5'
ICD-10-CM diagnosis code for Palliative Care. Applied via LIKE pattern '%Z51.5%' on claimdiag.codeid and authdiag.diagcode. Do not change without DHCS measure specification update.
@leadlag
decimal(9,0) = -4
Number of months to shift back from the 1st of the prior month. -4 = 4 months back. Determines start of the reporting quarter window. Adjust only for off-cycle or retrospective QA runs.
@leadlagmonths
int = 3
Duration of the reporting quarter in months. 3 = standard quarter. @RangeEndDate = @RangeStartDate + 3 months - 1 day. Do not reduce below 3 for standard quarterly submission.
@RangeStartDate
Derived via DATEADD
= DATEADD(MONTH, @leadlag, DATEADD(DAY, 1, EOMONTH(GETDATE(),-1))). Resolves to 1st of the month 4 months prior to the current run month. Used for Claims track pre-disqualify and qualify date filters.
@RangeEndDate
Derived via DATEADD
= DATEADD(day, -1, DATEADD(MONTH, @leadlagmonths, @RangeStartDate)). Last day of the 3-month window starting from @RangeStartDate. Used as upper bound for claim.startdate filter in #QUALIFYING_CLAIMS.
@ClockStart / @ClockStop
Same as Range dates
Auth track date window parameters. @ClockStart = @RangeStartDate, @ClockStop = @RangeEndDate. Used for referral.seendate BETWEEN filter and EnrollmentManager BenefitBeginDate / BenefitEndDate window.
@Authlob
nvarchar = 'DSNP'
LOB filter for the Auth/Referral track (SETAUTHREF CTE). Mirrors @LOB. Declared separately in the Auth parameter block for clarity — must match @LOB value used in the Claims track.
@when
datetime = NULL
Reserved parameter — currently SET to NULL in both scripts. Available for future targeted date overrides if off-cycle QA runs require a specific reference date. Do not set unless explicitly directed.
@Authrundt
TRY_CONVERT(date, GETDATE())
Auth track run timestamp — current date as of execution. Used for Auth-level date stamping in the SETAUTHREF output. Resolves dynamically — no override needed for standard quarterly runs.
