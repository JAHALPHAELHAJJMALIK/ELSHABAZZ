Exit Code Behavior

Exit 1 — Hard fail → SSA halts, Database Mail fires
Causes: INI config missing, month folder not found, any CHGSD_* prefix absent, STAGING create failed, ZIP creation failed.

Exit 0 — Success or soft fallback → SSIS steps proceed
MOVEit API failures are non-fatal. ZIP is staged regardless — MOVEit picks it up on its Friday 11:00 AM schedule even if the API trigger failed.

Exit 1 → DB Mail Exit 0 → SSIS proceeds API fail → ZIP staged

C — Manual Re-Run / Recovery (off-schedule or after failure)

  • Confirm source .dat files in the month folder are current before re-running before re-run
    Browse to \SYMPHONY\{YYYY}\{YYYY}_{Month}\ and confirm all 12 CHGSD_* prefixes are present with recent dates. If any are missing, SQLPRODAPP01 Schedule 22 may not have completed — wait for source refresh and then re-run.
  • Manually start SSA Job at Step 1 from SSMS recovery
    SSMS → SQL Server Agent → Jobs → Symphony dat File Upload → right-click → Start Job at Step → select Step 1. This creates a fresh ZIP and calls the MOVEit API. Monitor history for result.
  • If MOVEit API was unavailable — manually trigger MOVEit Task after ZIP is staged recovery
    When the PS script exits with a fallback warning, the ZIP is still staged. Either wait for the Friday 11:00 AM MOVEit schedule, or trigger manually:
    devops01 → Tasks → Symphony dat File Upload → Run Now
    Before running manually, verify: Source Path = STAGING UNC, File Mask = *.zip.
  • If MOVEit Task is Skipped (NewFilesOnly) — delete the STAGING ZIP and re-stage recovery
    MOVEit's NewFilesOnly flag prevents re-transferring a file it has already seen. To force a re-send: delete the ZIP from STAGING, re-run SSA Step 1 (which creates a fresh ZIP), then trigger the MOVEit Task manually.
  • If new month folder not yet created — script uses fallback to most-recent folder awareness
    The script first looks for \SYMPHONY\{YYYY}\{YYYY}_{MonthName}\. If not found, it scans subdirectories for the most-recently-modified year/month folder. The SSA Step 1 history output logs exactly which folder was selected — verify it is correct before accepting the run result.

Key Lesson (2026-05-15): Availity MFT uses FTPS Explicit on port 22 — not SFTP on port 22. MOVEit's FTP host type supports FTPS but the Secure Connection dropdown must be set to "Explicit (Ctrl/Data)". When in doubt: port 21 = FTPS, port 22 = SFTP.

Why CmdExec, not PowerShell step type? The SSA PowerShell step type uses a constrained runspace that blocked loading .NET types (caused [System.IO.Compression.ZipArchiveMode] error during C002→C003 migration). CmdExec + powershell.exe spawns a full unrestricted PowerShell 5.1 session. The -ExecutionPolicy Bypass flag is required because CHG\qnxtadmin may have a restricted execution policy on SQLPROD02.
Troubleshooting — Known Errors & Resolutions
Error Decision Trees

⚠ SSA Step 1 fails instantly — "The system cannot find the file specified"

1Open SSMS → SQL Server Agent → Jobs → Symphony dat File Upload → Steps → Step 1 → Edit. Check the Command field.
2Confirm the command begins with powershell.exe (not python.exe). Python is not installed on SQLPROD02 and cannot be added without admin rights.
3Confirm the -File path is enclosed in double quotes and uses the correct UNC path. UNC paths in SSA commands require regular backslashes (not forward slashes).
4Verify the .ps1 file exists at the UNC path: open Windows Explorer and browse to \\netapp02\IS\Informatics\_Informatics Deliverables\Operations\Projects\SYMPHONY\.

⚠ SSA Step 1 fails — "Config file not found: symphony_moveit_upload.ini"

1The script looks for symphony_moveit_upload.ini (no date suffix) in its own directory. The desktop file has a date suffix; the deployed file must not.
2Copy the INI to netapp02 and rename to exactly: symphony_moveit_upload.ini
3Verify the deployed INI contains correct UNC paths and MOVEit credentials (Configuration tab).

⚠ SSA Step 1 fails — "Missing .dat files for N prefix(es): CHGSD_*"

1One or more of the 12 CHGSD_* prefixes has no matching .dat file in the resolved month folder.
2Check SSA Step 1 history output for the "Resolved current folder" log line. Confirm the correct month folder was scanned.
3Browse to that folder. If a prefix is missing, SQLPRODAPP01 Schedule 22 may not have completed yet. Wait and re-run.
4If it is a new month and the new folder does not exist yet, the script logs a fallback folder. Confirm the fallback folder's contents are acceptable for this run.

⚠ SSA Step 1 fails — "Unable to find type [System.IO.Compression.ZipArchiveMode]"

1This means an older version of the script (which used System.IO.Compression) was deployed. The current C003 version uses Compress-Archive instead.
2Open the deployed .ps1 and search for "ZipArchiveMode" — it should not appear. The correct version uses Compress-Archive -Path $filePaths -DestinationPath $zipPath.
3If ZipArchiveMode is present, redeploy symphony_moveit_upload_20260515.ps1 (C003) from the desktop.

⚠ SSA Step 1 succeeds (exit 0) but history shows "MOVEit Task not found by name"

1This is a soft fallback — Step 1 exits 0, SSIS steps still run, ZIP is staged. Not a hard failure. MOVEit Task will run on its Friday 11:00 AM schedule.
2Log in to devops01. Confirm the Task is named exactly Symphony dat File Upload (spelling must match; case is ignored).
3Confirm the INI [moveit] task_name value matches. Update INI if the task was renamed.
4If the task does not exist on devops01, clone any existing task, rename it, and configure source/destination per the Configuration tab.

⚠ MOVEit FTP Host test fails — "hostname or IP address is not valid"

1The Host/Address field contains a full URL. devops01 → Hosts → Symphony Availity QA → Edit.
2Change the Host field to exactly: files.availity.com (no https://, no trailing slash, nothing else).
3Save and re-test. Expect "Connection successful."

⚠ MOVEit FTP Host connection fails — port 22 / SFTP connection refused

1Availity MFT uses FTPS (FTP over TLS), not SFTP (SSH). Port 22 is the wrong protocol.
2devops01 → Hosts → Symphony Availity QA → Edit. Set Port = 21. Set Secure Connection = Explicit (Ctrl/Data).
3Do NOT use the SFTP/SSH host type. Use the FTP host type with Explicit TLS.
4Save and re-test. Expect "Connection successful."

⚠ MOVEit Task runs but no file appears in Availity QA inbox

1Check MOVEit Task log: devops01 → Dashboard → Recent Task Activity → Symphony dat File Upload. Confirm status = Completed, not Skipped or Failed.
2If Skipped: NewFilesOnly=1 means the Task found no new files. The same ZIP was already transferred. Delete the STAGING ZIP, re-run SSA Step 1, then trigger MOVEit manually.
3If Completed with 0 files: STAGING was empty when the Task ran. Confirm the ZIP was created in STAGING before the Task executed.
4If Completed and MOVEit log shows the file was sent: check the Availity Received/Archived tab. The file may be auto-archived from the inbox view.
