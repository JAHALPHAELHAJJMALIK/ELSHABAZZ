Alignment Review — Findings & Analyst Controls
Finding — ECM Modifier Logic (Fixed 20260710): The baseline code's ECM CASE branch checked modcode IN ('U1') OR modcode2 IN ('U2') OR modcode3 IN ('U8') — i.e. it required U1 specifically in slot 1, U2 specifically in slot 2, U8 specifically in slot 3. Real claims populate their (usually single) modifier in slot 1 (modcode) regardless of its value, so a claim billed with modifier U2 or U8 would almost never satisfy this condition. This directly conflicts with Ms. Claudia's 7/9/2026 note that unsuccessful-outreach claims for ECM-eligible members should not be excluded, and with Mr. Rodrigo Diaz's 7/10/2026 instruction to include U1, U2, and U8 as qualifying modifiers.
Fix Applied (C004): Corrected to (modcode IN ('U1','U2','U8') OR modcode2 IN (...) OR modcode3 IN (...) OR modcode4 IN (...) OR modcode5 IN (...)) — checks all 5 modifier slots for any of the three qualifying values. This is the exact pattern already used for every CS service type in this same script and in the STEP88 CS reference script, so the fix brings the ECM branch into consistency with the rest of the codebase, not just with the ask.
Interim post-C004 execution, 7/10/2026 (superseded below): DUP validation returned 0 rows. QA pre-stage counts at that point: CS = 68,391 unique claims / 8,658 unique members; ECM = 16,663 unique claims / 7,597 unique members. QA LOB(S) confirmed CSNP, DSNP, MEDI-CAL, and MSO-CHPIV all present. Total Paid (still blended by [Cx]): $23,453,198.28. This run predates the C005 rollup fix below and the numbers moved substantially once ECM/CS were correctly split — see the validated figures.
Finding — Cost-per-Claim Rollup Silently Blends ECM + CS (Fixed 20260710, C005): The final "BASELINE SUMMARY BY LOB" query — the one that actually computes Cost per Claim = SUM(PAID_NET_AMT) / COUNT(DISTINCT claimid) and is the closest thing in this script to Ms. Claudia's original "average cost spent on ECM and CS" ask — grouped by ISNULL(su.Cx,'NOT YET CLASSIFIED'). The [Cx] column is declared NULL at the top of the script and never subsequently set to anything else, so every row falls into the same 'NOT YET CLASSIFIED' bucket, meaning ECM and CS paid amounts were combined into one blended figure per LOB, not reported separately.
Fix Applied (C005): Added su.[InclusionExclusionType] to both the SELECT list and the GROUP BY of the BASELINE SUMMARY BY LOB query. [InclusionExclusionType] is already populated upstream ('ECM Service Received' / 'CS Service Received') and carried through the whole pipeline — it was simply never wired into this final rollup. Billed/Allowed/Net/Paid/TRI Payment/Cost per Claim now report separately for ECM vs CS within each LOB + Primary/Secondary Status combination.
Validated final run, 7/10/2026 (C004 + C005 both in effect): CS Service Received = 48,734 paid claims / $20,431,555 paid. ECM Service Received = 112,852 paid claims / $45,327,497 paid. Combined = 161,586 paid claims / $65,759,051 paid — roughly 2.8x the earlier blended $23.45M figure. This magnitude increase is expected and correct: the pre-C004 modifier bug was severely undercounting ECM (checking the wrong modifier slot for U2/U8), so the true ECM-paid figure was always closer to $45.3M than the $23.45M blended total ever implied. No ECM rows appear under CSNP, MSO-CHPIV, or DSNP-Primary — confirmed by Walter Carr as expected, since ECM is a Medi-Cal-only benefit that should not apply to Medicare-side coverage.
