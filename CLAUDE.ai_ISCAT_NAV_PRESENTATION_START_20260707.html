ISCAT Section VI — Integration & Control of Data for Network Adequacy Reporting
How Network Adequacy Reports Are Named, Stored & Retained
Standardized approach applied to every SQIC/CQIC network adequacy indicator monitoring report, cited directly in ISCAT NAV readiness responses.
ComponentStandard
File namingReport type + LOB + reporting period + version, e.g. SQIC_MCAL_PCP_AccessAnalysis_yyyymmdd.xlsx, SQIC_DSNP_Maps_yyyymmdd.pdf
Primary storagefile://fileserver01/Corporate Shared Drive/Information Systems/Informatics/WCarr/NETWORK/ — subfoldered by reporting period, LOB, and report type; access restricted to authorized personnel
Archiving & retentionHistorical reports retained per data-retention policy in zipped, year-organized archive folders to support audit trails and historical comparison
Version controlVersion numbers appended to file names; change log maintained documenting the nature/purpose of each revision
Regulatory submissionDHCS-bound reports stored in a dedicated submission folder labeled by submission date + regulatory period, e.g. Meetings/Regulatory Reports/Subcontracted Network Certification/{year}/Resubmission/
Reporting cycleQuarterly (generated first week of the reporting quarter) and Annual, following the same generation/storage approach
Example archived report family: SQIC_DSNP_AccessAnalysis_yyyymmdd.pdf · SQIC_DSNP_Maps_yyyymmdd.pdf · SQIC_MediCal_BehavioralHealth_Providers_AccessAnalysis_yyyymmdd.pdf · SQIC_MediCal_BehavioralHealth_Providers_Maps_yyyymmdd.pdf · SQIC_MCAL_FAC_AccessAnalysis_yyyymmdd.pdf · SQIC_MCAL_PCP_AccessAnalysis_yyyymmdd.pdf · SQIC_MCAL_SPE_AccessAnalysis_yyyymmdd.pdf · CQIC_Threshold_Languages_AccessAnalysis_yyyymmdd.pdf · Provider_Network_Analysis_Language_Race_Ethnicity_yyyymmdd.xlsx
Why it matters for NAV: ISCAT Section VI asks how data merges are controlled and how reasonableness is checked month-over-month. A consistent naming/storage/retention convention is the paper trail an auditor uses to confirm reports weren't altered after generation and that the correct version was the one submitted to DHCS.
Related ISCAT Narrative — Data Sources & Integration (Section I.E / VI, as filed)
Enrollment System

Primary source of member demographic, plan enrollment, and contact data; regularly updated and maintained for accuracy.

Provider Data Management System (eVIPS)

Comprehensive provider info — specialties, locations, contact detail, panel capacity — updated through contracting, credentialing, and verification.

Claims Processing System

Utilization data (date of service, provider, service detail), including Telehealth network claims, supporting access-issue identification.

ETL → Data Warehouse → Data Mart

Automated ETL validates, cleanses, and loads source data into a centralized warehouse (star schema); network-adequacy-specific data marts are pre-aggregated for reporting performance.

Geocoding → Time/Distance → Provider Capacity

Member/provider addresses geocoded; time/distance computed via Quest Analytics; panel capacity analyzed to surface network gaps before they reach DHCS.

Supervisory Review & Continuous Monitoring

Designated staff review reports for accuracy/completeness/regulatory adherence prior to finalization; audits and feedback loops drive ongoing improvement.
